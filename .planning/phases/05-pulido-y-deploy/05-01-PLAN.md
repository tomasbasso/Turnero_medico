<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cliente anónimo → POST /api/auth/login | Input no autenticado; el rate limiter corre ANTES de cualquier auth (no hay sesión todavía) |
| cliente anónimo → POST /api/public/appointments | Input público sin registro; cualquiera puede llamar |
| proxy/CDN → header x-forwarded-for | El identificador de rate limit depende de un header que el cliente/proxy puede manipular |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-01 | Spoofing | POST /api/auth/login | mitigate | `loginLimiter.limit(ip)` con sliding window 5/15min por IP; 429 sin filtrar si el email existe (D-12) |
| T-5-02 | Denial of Service | POST /api/public/appointments | mitigate | `bookingLimiter.limit(ip)` con sliding window 10/10min por IP; 429 con mensaje genérico |
| T-5-03 | Tampering | header x-forwarded-for | accept | Spoofeable en proxies mal configurados; aceptable para un consultorio (no es banco). `getIp` usa la primera IP de la lista para minimizar colisiones (Pitfall 5) |
| T-5-04 | Denial of Service | arranque del servidor sin env vars Upstash | mitigate | Guarda `if (hasUpstash)` evita instanciar `Redis.fromEnv()` sin credenciales (Pitfall 2); fallback in-memory permite `npm run dev` local |
| T-5-SC | Tampering | npm install (@upstash/ratelimit [ASSUMED]) | mitigate | Checkpoint humano bloqueante (Task 1) verifica publisher en npmjs.com/github antes de instalar |
</threat_model>
